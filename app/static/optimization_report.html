Optimizations tested and applied

  1. appliedDisable hidden reasoning: thinkingConfig.thinkingBudget = 0 for the Gemini 2.5 family. These models spend "thinking" tokens by default — useless for a classification task. Probe: 2.5-flash 2 926 → 870 ms per raw call. This one flag is what turned 2.5-flash from timing out into a viable tier.
  2. appliedCompact wire schema + output cap: the model replies with single-letter JSON keys ({"i","l","p","t","c"}, mapped back to full names server-side) under maxOutputTokens: 96. Decode time dominates generation, so halving output tokens is the cheapest speedup. Combined with a ~60% smaller prompt (terse rules, inline few-shot examples, user-context profile as one line): flash-lite p50 683 → 372 ms.
  3. appliedRegional Vertex endpoint: 2.5 models are served from europe-west4 (adjacent to the Cloud Run region) instead of the global endpoint — probed ~250 ms RTT saving warm. Per-model routing, override via VERTEX_LOCATION.
  4. appliedTTL response cache keyed by (model, normalized query, user context): identical repeats cost 0 ms. Deliberately context-aware — the same user's repeat after their interest profile changed is a different key (personalization correctness over hit rate).
  5. appliedRobustness that protects latency: responseMimeType: application/json + temperature 0 (no parse retries), string "null" outputs normalized, hard 6 s timeout + one retry with fallback to the rules answer (LLM_TIMEOUT_S tunable) — the API never blocks on a slow model.

Tested and rejected

  1. rejectedthinkingBudget: 0 on 3.x models: probed no effect on 3.5-flash (3 122 → 3 226 ms) and slightly worse on 3.1-flash-lite; applied to the 2.5 family only.
  2. rejectedRegional endpoint for 3.x models: probed 404 on europe-west4/west1 — these models are global-endpoint-only.
  3. rejectedgemini-3.5-flash as a serving tier: its inference floor (~3 s+, plus mandatory thinking that the output cap truncates) cannot meet the 6 s budget; it fast-fails to the deterministic answer. Kept in the allowlist only for benchmarking honesty.
  4. rejectedStreaming: the agent needs the complete JSON before acting, so time-to-first-token gains don't translate to end-to-end latency here.

Architecture-level context

The strongest optimization remains not calling the LLM: a learned query classifier + concept-grouped BM25 answer every known-vocabulary query in ~1–2 ms, the LLM runs only on unknown terms (or on request via llm_mode: always), and recurring misses become catalog tags or classifier training data — the escalation rate shrinks over time.
